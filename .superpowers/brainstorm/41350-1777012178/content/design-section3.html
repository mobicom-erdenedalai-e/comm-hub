Design — Section 3: Error Handling & Testing

How the system fails gracefully and how we verify it works

ERROR HANDLING
📡
Source API unavailable (GitHub / Jira / Slack)
Skip that source, continue with what's available. Generated draft notes which sources were used (e.g. "Based on GitHub and Jira — Slack data unavailable"). Dev decides whether to regenerate later.
🤖
GitHub Models API failure
Show inline error with a Retry button. Never show a blank screen. If retry fails twice, surface a plain "AI unavailable — try again later" message.
🔐
Expired or missing credentials
Detected at startup and on each connector call. Show a clear banner linking to the Settings page to reconnect — never silently fail.
📭
No activity data in date range
Prompt the dev before calling AI: "No activity found for this period — generate anyway?" Prevents blank or nonsense reports.
TESTING STRATEGY
🧪 Unit Tests
  • Each connector's data normalization
  • Aggregator merge logic
  • Prompt template rendering
  • Client tone injection
🔗 Integration Tests
  • GitHub API connector (sandbox repo)
  • Jira API connector (test project)
  • GitHub Models API (real call, small prompt)
  • DB read/write for artifacts
🖥️ E2E Tests
  • Login via GitHub OAuth
  • Add client + connect Jira
  • Generate weekly report → copy
  • Upload transcript → summary
