Design — Section 2: Components & Artifact Generation

How the system is structured internally — does this make sense?

COMPONENT MAP
📡 Connector Layer
One module per source: github.ts, jira.ts, slack.ts, meeting.ts
Each exposes a single fetch(clientId, dateRange) function that returns normalized activity items.
🗂️ Aggregator
Calls all connectors in parallel. Merges results into a single ActivityBundle — a structured summary of what happened in the date range.
✍️ Prompt Engine
One prompt template per artifact type. Injects the ActivityBundle + client tone settings. Sends to GitHub Models API and returns the raw draft string.
💾 Storage
PostgreSQL via Prisma ORM. Tables: clients, integrations, artifacts (generated history), users.
ARTIFACT PROMPT TEMPLATES
📄 Weekly Report
Inputs: commits, merged PRs, closed Jira tickets, sprint velocity. Output: 3-section email (Done, In Progress, Next Week).
🗒️ Meeting Summary
Input: raw transcript text. Output: bullet-point summary with decisions made, action items (owner + deadline), and open questions.
💬 Status Reply
Input: client's question (typed by dev) + latest activity bundle. Output: a concise, professional reply paragraph.
📦 Handover Doc
Inputs: full commit history, all closed tickets, open issues. Output: structured markdown doc with architecture summary, deployment steps, and known issues.
PER-CLIENT TONE SETTINGS
Each client config stores: tone (formal / friendly / technical), preferred language (English/other), and report format (email prose vs bullet points). These are injected into every prompt for that client so the output always matches their expectations.
